# Section 7 — AI Module Development

**Project:** Cognitive Business Automation System (CBAS)
**Module:** AI Engine — lead scoring, churn prediction, sentiment analysis, recommendations, and an intelligent chatbot.

---

## 7.1 Overview and Design Rationale

The CBAS AI module is delivered as a dedicated **AI Engine**: a standalone
Python/FastAPI service that trains and serves the machine-learning models over a
REST API, which the main Next.js application calls and stores results against CRM
records. This separation keeps the model code isolated, independently testable,
and off the Node.js runtime, matching the architecture proposed for the system.

No large language model is trained from scratch. Four of the five features are
classic supervised ML / NLP models trained on tabular or text data; the fifth
(chatbot) integrates a pretrained hosted LLM through retrieval-augmented
generation. This scope is realistic, defensible, and runs on free compute.

Every model follows one repeatable pipeline: load and clean the dataset, encode
features, split 80/20 (stratified), train a simple baseline, train an improved
model, evaluate on a held-out test set, and persist the model with `joblib`
behind a FastAPI endpoint.

| # | Feature | Problem type | Baseline → Improved | Endpoint |
|---|---------|--------------|---------------------|----------|
| 1 | Lead Scoring | Binary classification | Logistic Regression → **XGBoost** | `POST /score-lead` |
| 2 | Churn Prediction | Binary classification | Random Forest → **XGBoost** | `POST /predict-churn` |
| 3 | Sentiment Analysis | Text classification (3-class) | **TF-IDF + Logistic Regression** | `POST /analyze-sentiment` |
| 4 | Recommendations | Recommender system | **Truncated-SVD collaborative filtering** | `GET /recommendations/{id}` |
| 5 | Chatbot | Conversational AI | **Gemini LLM + RAG retrieval** | (in-app, `askAssistant`) |

All results below are on **real Kaggle datasets** (held-out test sets). Figures
referenced are generated by `ai-engine/scripts/make_figures.py` into
`ai-engine/artifacts/figures/`.

## 7.2 Lead Scoring

**Purpose.** Predict whether a lead will convert and rank leads by predicted
probability so the sales team works the highest-value leads first.

**Data.** X-Education Lead Scoring dataset (9,240 leads, 38.5% conversion rate).
Features: lead origin and source, total visits, time on site, page views per
visit, last activity, current occupation, and specialization. Target:
`Converted`. The dataset's "Select" placeholders are treated as missing and
imputed (median for numeric, most-frequent for categorical).

**Method.** A Logistic Regression baseline (interpretable) is compared against an
XGBoost model; the model with the higher ROC-AUC is kept. Class imbalance is
handled with class weighting / `scale_pos_weight`.

**Results (XGBoost, held-out test).**

| Metric | Score |
|--------|-------|
| ROC-AUC | **0.876** |
| Accuracy | 0.812 |
| Precision | 0.732 |
| Recall | 0.809 |
| F1 | 0.769 |

Confusion matrix `[[925, 211], [136, 576]]` — of 712 true converters, 576 are
caught (high recall, appropriate for prioritising follow-ups). The strongest
features are the *Lead Add Form* origin, *Working Professional* occupation, *SMS
Sent* last activity, and the *Reference* source.

**Figures.** `roc_lead_scoring.png`, `cm_lead_scoring.png`,
`importances_lead_scoring.png`.

**Integration.** `scoreLead()` maps a CRM lead onto the model's features, writes
back `aiScore` (0–100), `aiScoreBand` (Hot/Warm/Cold) and a rationale; a score
card + **Score** button appear on the lead detail page.

## 7.3 Churn Prediction

**Purpose.** Flag customers likely to stop buying so the business can intervene
early with retention offers.

**Data.** Telco Customer Churn dataset (7,043 customers, 26.5% churn rate).
Features: tenure, monthly and total charges, contract type, payment method,
internet service, paperless billing, tech support, online security, senior-
citizen flag. Target: `Churn`. Blank `TotalCharges` values (new customers) are
coerced to numeric and imputed.

**Method.** Random Forest baseline vs. XGBoost; best kept by ROC-AUC.

**Results (XGBoost, held-out test).**

| Metric | Score |
|--------|-------|
| ROC-AUC | **0.839** |
| Accuracy | 0.750 |
| Precision | 0.519 |
| Recall | 0.767 |
| F1 | 0.619 |

Confusion matrix `[[769, 266], [87, 287]]`. Recall on churners (0.767) is
deliberately favoured — missing an at-risk customer costs more than a false
alarm. The dominant driver is a **month-to-month contract**, followed by fibre-
optic internet and the absence of online security / tech support — consistent
with the established Telco churn literature.

**Figures.** `roc_churn.png`, `cm_churn.png`, `importances_churn.png`.

**Integration.** The Telco model is retained as an external **benchmark**. In the
running app, churn is served by a **CRM-native model trained on CBAS's own data**
(see §7.9) — `predictContactChurn()` engineers engagement features from the
contact's orders, activities and deals and calls that model, writing a
Low/Medium/High band to the `Contact` record; the contacts table shows a **Churn
Risk** column and a **Predict Churn Risk** action.

## 7.4 Sentiment Analysis

**Purpose.** Read customer reviews / feedback and label them positive, negative,
or neutral to feed a satisfaction signal.

**Data.** Product-review dataset (137,102 labelled texts after cleaning; classes:
positive 22,224 / negative 9,224 / neutral 2,828 in the test split). Text is
lower-cased and stripped of punctuation.

**Method.** The plan's baseline — TF-IDF vectorisation (uni- and bi-grams,
English stop-words, sublinear TF) followed by a class-weighted Logistic
Regression, three-way classification.

**Results (held-out test).**

| Metric | Score |
|--------|-------|
| Accuracy | **0.844** |
| Macro-F1 | 0.721 |

| Class | Precision | Recall | F1 |
|-------|-----------|--------|-----|
| Positive | 0.918 | 0.925 | 0.921 |
| Negative | 0.898 | 0.747 | 0.815 |
| Neutral | 0.357 | 0.531 | 0.427 |

Positive and negative are classified strongly; neutral is the hardest class
(genuinely ambiguous text and far fewer examples) — a known characteristic of
three-class review sentiment and an honest limitation to note in the defence.

**Transformer upgrade (DistilBERT).** As the plan anticipates, a pretrained
**DistilBERT** was fine-tuned (transfer learning, not trained from scratch) via
`training/sentiment_bert.py`, on an NVIDIA Quadro P1000 GPU (48,000 training
reviews, 2 epochs, mixed precision). Evaluated on the **same natural
distribution** as the baseline for a like-for-like comparison, it **beats the
TF-IDF baseline on both metrics**:

| Model | Accuracy | Macro-F1 | Positive F1 | Negative F1 | Neutral F1 |
|-------|----------|----------|-------------|-------------|------------|
| TF-IDF + LogReg | 0.844 | 0.721 | 0.921 | 0.815 | 0.427 |
| **DistilBERT (GPU)** | **0.859** | **0.731** | 0.929 | 0.825 | 0.438 |

DistilBERT lifts accuracy from 84.4% to **85.9%**. Notably, even the transformer
finds the **neutral** class hard (F1 ~0.44) — confirming that class is genuinely
ambiguous, not a modelling deficiency. The lightweight TF-IDF model remains the
default served endpoint (no GPU dependency at inference); DistilBERT is available
as the higher-accuracy option (`artifacts/sentiment_bert/`,
`sentiment_bert_metrics.json`).

**Figure.** `cm_sentiment.png`.

**Integration.** When a note/activity is logged, `createActivity` calls the model
and stores `sentiment` + `sentimentScore`; a coloured badge renders beside the
note in the lead timeline. It is best-effort, so notes still save if the engine
is offline.

## 7.5 Personalized Recommendations

**Purpose.** Suggest products per customer from purchase history.

**Data.** Online Retail transactions (267,615 interactions, 4,372 customers,
3,684 products). A user-item matrix is built from purchase quantities
(`log1p`-scaled implicit feedback).

**Method.** Collaborative filtering by **matrix factorization** — Truncated SVD
(32 latent factors) over the user-item matrix. A user's affinity for an item is
the dot product of their latent vectors; already-purchased items are excluded.
Unknown customers receive a popularity-based cold-start fallback.

**Results (leave-out evaluation).**

| Metric | Score |
|--------|-------|
| Precision@5 | **0.149** |
| Recall@5 | 0.086 |

These are realistic for sparse retail data with thousands of items (a random
baseline is ≈ 0.001). Qualitatively, a real customer (ID 17850) is recommended
coherent related items ("hand warmer", "wicker heart", "t-light holder").

**Integration.** The Online Retail model is retained as a **benchmark**. The app
serves a recommender **retrained on CBAS's own `Order`/`OrderItem` history** (see
§7.9), so `getContactRecommendations()` returns personalized picks using the
app's real products and customers; a **Recommend Products** action on each
contact shows them.

## 7.6 Intelligent Chatbot (LLM + Retrieval)

**Purpose.** Answer questions about the business and its CRM records, and perform
confirmable actions.

**Method.** The plan's *Path B* — a hosted LLM (Google Gemini) with
**retrieval-augmented generation**. Workspace records (leads, contacts, deals)
are embedded with a local sentence-transformer (`all-MiniLM-L6-v2`) and cached by
record id + `updatedAt`; for each question the top-k most semantically similar
records are retrieved (cosine similarity) and injected into the prompt, so the
assistant answers about **specific** records rather than aggregate counts. The
model may also propose one of two whitelisted tools (log expense, receive stock),
which execute only after explicit user confirmation.

**Validation.** For the query *"Which leads are most likely to convert?"* the
retriever scored a qualified, high-AI-score lead at **0.33** vs. **0.04** for an
unrelated lost deal — correctly surfacing the relevant record.

**Integration.** `askAssistant()` (`src/lib/actions/chatbot.ts`) +
`retrieveContext()` (`src/lib/ai/retrieval.ts`). Requires `GEMINI_API_KEY`.

## 7.7 Architecture and Deployment

```
Frontend (Next.js / React)
        │
Backend (Next.js server actions)  ──HTTP──▶  AI Engine (FastAPI, Python)
        │                                          │
     PostgreSQL                           Trained models (joblib)
                                                   │
                                          Gemini API (chatbot, Path B)
```

Endpoints: `POST /score-lead`, `POST /predict-churn`, `POST /analyze-sentiment`,
`GET /recommendations/{id}`, plus `/health` and `/models/*` for metrics. An
optional `x-ai-engine-key` header secures the service. Training and serving are
reproducible: `scripts/fetch_datasets.py` pulls all four datasets and
`scripts/train_all.py` retrains every model.

## 7.9 App-native Models (trained on CBAS's own data)

The benchmark models above validate each technique on established public
datasets. For the two features whose problem domain does not match a CRM/BAS
schema — churn (Telco is telecom-specific) and recommendations (the retail item
space is not the app's products) — a **second, app-native model** is trained
directly on CBAS data so it runs on real customers and products.

**Data.** Because a newly-deployed system has little history (a cold-start
problem), an app-native dataset is generated *inside the CBAS schema*
(`scripts/seed-ai-demo.ts`): 48 products, 400 contacts each with a latent taste
and engagement profile, 1,596 orders (3,079 line items), and 1,422 time-spread
activities with sentiment. `scripts/export-ai-training.ts` exports this to
`crm_orders.csv` and `crm_churn.csv`. The customer *behaviour* is synthetic, but
it lives in the real schema (`Order`, `OrderItem`, `Activity`, `Contact`,
`Product`), so the models learn from the app's actual entities. As real usage
accumulates, the same export + train commands produce genuinely data-driven
models.

**Churn (CRM-native).** A customer is labelled *lapsed* when their most recent
activity is older than a 60-day window; the model predicts this from engagement
features — tenure, order count, spend, activity count, note sentiment, product
breadth and open-deal state — with the label-defining recency deliberately
excluded. XGBoost (vs. Random Forest baseline) on 453 contacts:

| Metric | Score |
|--------|-------|
| ROC-AUC | 0.972 |
| Accuracy | 0.934 |
| F1 | 0.941 |
| Recall | 0.941 |

The strongest predictors are **average note sentiment (0.31)**, **activity count
(0.24)** and **tenure (0.15)** — engagement signals, not the recency that defines
the label, confirming the model is not learning the label circularly. The high
scores partly reflect the clean latent structure of the generated data and would
moderate on messier real-world data; the value here is a model that consumes the
app's real features end-to-end. Figures: `roc_churn_crm.png`, `cm_churn_crm.png`,
`importances_churn_crm.png`.

**Recommendations (CRM-native).** The same Truncated-SVD collaborative filter,
retrained on the app's `Order`/`OrderItem` history (382 customers, 48 products):
Precision@5 0.044, Recall@5 0.161. It returns real product names for real
contacts and is served by the app's **Recommend Products** action.

**Combined design.** Both benchmark and app-native artifacts are kept side by
side (`churn.joblib` + `churn_crm.joblib`; `recommend.joblib` (app) +
`recommend_kaggle.joblib` (benchmark)). The report cites the benchmark to prove
the technique on real external data, and the app-native model to show the
integrated system working on CBAS's own database.

| Feature | Benchmark (external data) | App-native (CBAS data) |
|---------|---------------------------|------------------------|
| Churn | Telco, ROC-AUC 0.840 | CRM engagement, ROC-AUC 0.972 |
| Recommendations | Online Retail, P@5 0.149 | App orders, P@5 0.044 (48 items) |
